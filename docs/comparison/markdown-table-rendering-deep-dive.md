# Qwen Code 改进建议 — Markdown 表格终端渲染 (Markdown Table Rendering)

> 核心洞察：当代码 Agent 汇报任务总结、对比多个库的特性或者展示性能数据时，Markdown 表格是最直观的表达方式。但在纯文本的终端模拟器中，如果不对表格进行复杂的“列宽计算与等宽对齐”，渲染出来的表格就会错位、破裂，完全不可读。特别是涉及到中文字符（CJK 双宽字符）和带有 ANSI 高亮颜色的文字时，简单的 `string.length` 会导致对齐全盘崩溃。Claude Code 内置了极其强悍的 ANSI-Aware 和 CJK-Aware 表格排版引擎，而 Qwen Code 目前在终端呈现中英文混合表格时，表现差强人意。
>
> 返回 [改进建议总览](./qwen-code-improvement-report.md)

## 一、终端表格排版的三座大山

### 1. Qwen Code 现状：排版破裂
在当前的 TUI 组件（如 `MarkdownDisplay.tsx`）中，当大模型输出如下包含表格的 Markdown 时：
```markdown
| 组件名 | 状态 | 错误描述 |
| --- | --- | --- |
| \x1b[31mHeader\x1b[0m | Failed | 渲染越界 |
```
Qwen Code 会直接将其交给通用的 Markdown 渲染器或简单转义后丢给终端。
- **痛点一（ANSI 幽灵长度）**：对于带有颜色的词 `\x1b[31mHeader\x1b[0m`，肉眼看到的是 6 个字符长。但底层的 `string.length` 可能会算成 15 个字符。如果在绘制表格的竖线 `|` 时使用补齐空格，它会多补 9 个空格，导致这一行的竖线被狠狠地向右推了出去。
- **痛点二（中日韩双列宽）**：在终端网格（Grid）系统中，一个英文字母占 1 列宽，而一个中文字符“中”占 2 列宽。如果 `length` 算出来是 1，补齐时也会错位，导致中文表格惨不忍睹。
- **痛点三（无自动折行）**：如果某一列的文本太长（如很长的报错堆栈），它会直接顶破终端的物理右边界，甚至引发终端滚动的错乱。

### 2. Claude Code 解决方案：高精度测宽与自动重排
Claude Code 的 `components/MarkdownTable.tsx` 为了解决这个排版顽疾，可以说是煞费苦心。

#### 机制一：智能测宽器 (stringWidth)
它没有使用简单的 `.length`，而是借助了原生优化的测宽算法（如 `Bun.stringWidth` 或专门剥离 ANSI 的 `wcwidth` 库）。
在绘制表格前，系统会先在内存中遍历整张表的所有单元格：
1. 剥除所有 `\x1b` 颜色控制符。
2. 精确计算每个 CJK 字符为 2 个宽度，Emoji 为 2 个或多个宽度。
3. 统计出每一列所需的**真实物理最大列宽**。

#### 机制二：智能折行 (Cell Wrapping)
当所有列的物理宽度总和大于当前终端的实际宽度（通过 `process.stdout.columns` 动态获取）时，它不会任由表格顶破屏幕！
它会自动挑选最宽的列进行安全折行（Word Wrap），并在下一行中自动为其他短列补充空白填充（Padding），保证所有的竖线 `|` 笔直如尺。

#### 机制三：对齐标签支持
完美支持 Markdown 的 `:---` (左对齐)、`---:` (右对齐) 和 `:---:` (居中对齐) 语法。渲染引擎会根据对齐方式，灵活分配前后的填充空格。

## 二、Qwen Code 的改进路径 (P2 优先级)

对于定位于国内开发者的工具，对中文双宽字符的支持与表格呈现，是体现工具“精致感”的第一门面。

### 阶段 1：引入精密的宽字符计算依赖
1. 在 `packages/cli` 或者 TUI 渲染层，确保引入 `string-width` 和 `strip-ansi` 库。
2. 任何涉及到计算补齐空格 `padEnd` 的地方，必须使用 `stringWidth(text)` 而非 `text.length`。

### 阶段 2：重写 Markdown Table 渲染器
1. 在解析 Markdown 的 AST 树（如果是用 `marked` 或类似库）时，拦截 `table` 节点。
2. 编写一个渲染逻辑：
   - 第一遍循环（测宽）：找出每列所需的真实宽度。
   - 检查总宽是否超过终端列数。如果是，缩减最大列的配额。
   - 第二遍循环（绘制）：根据每列配额，对超长文本应用 `wrap-ansi` 进行多行截断，并用带颜色的竖线 `│` 进行拼接。

### 阶段 3：特殊字符降级
如果检测到表格内包含了极端复杂的转义或无法处理的 Unicode 组合，可以提供一个回退模式（Fallback）：取消边框绘制，退化为简单的行级纯文本输出，避免终端卡死。

## 三、改进收益评估
- **实现成本**：中等。实现带有折行逻辑和颜色保留的终端表格引擎有一定的代码复杂度，需要细致的单元测试，代码量在 150 - 250 行。
- **直接收益**：
  1. **极佳的数据展现力**：当输出模型对比、文件列表、重构状态等信息时，完美的表格对齐能给用户带来极大的心理舒适感。
  2. **完善的中文体验**：彻底根治中文、Emoji 导致终端界面崩坏的历史遗留问题。
